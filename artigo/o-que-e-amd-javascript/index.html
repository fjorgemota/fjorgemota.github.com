O que é AMD? (JavaScript)

Quando você vai implementar um grande web app, é comum possuir uma grande quantidade de arquivos JavaScript. Naturalmente (à menos que você seja louco em botar tudo em um só arquivo), os arquivos estabelecem grande relação de dependência entre si, sendo necessário que seja estabelecido, em algum lugar, o carregamento dos módulos na ordem correta. E, dependendo do tamanho da aplicação, isso pode ficar muito trabalhoso e ocasionar muitos problemas, e é por causa disso que hoje vamos responder a seguinte pergunta: O que é AMD? E dica: Não estamos falando sobre a fabricante de processadores.

AMD é um jeito de definir módulos em JavaScript. Por módulos, entenda arquivos que devem realizar apenas uma função ou grupo de funções do mesmo gênero (pois compartilham um mesmo escopo).

Considere, por exemplo, que temos um arquivo chamado hello.js com esta única função:

function hello(name){
    alert("Hello, "+(name||"Guest")+"!");
}

Usando AMD, o arquivo ficaria assim:

define(function(){
    function hello(name){
        alert("Hello, "+(name||"Guest")+"!");
    }
    return hello;
});

Note que a primeira vantagem que o uso de AMD em seus arquivos é o escopo: Os arquivos não tem acesso inicial ao escopo global, ficando emglobados dentro de uma mesma função, assim como imposto pelos controllers do Angular.js, do qual falamos ontem.

Além disso, digamos que nós temos um arquivo chamado foo.js que precisa da função hello. Sem AMD, o arquivo chamaria a função assim:

var userName = prompt("What's your name?");
hello(userName);

Hmmm..Meio confuso, né? E se a função simplesmente não existir ou não tiver sido carregada ainda?

Com AMD, a história é diferente:

define(["hello"], function(hello){
    var userName = prompt("What's your name?");
    hello(userName);
});

Note que no código acima, o arquivo especifica como requerimento o módulo hello.js (a extensão .js é sub-entendida no AMD), e recebe como parâmetro justamente o resultado retornado pelo módulo. Isso possibilita uma enorme quantidade de possibilidades:

  • Carregamento de módulos recursivos - Se o módulo hello possuisse alguma outra dependência, seria possível ao carregador AMD carregar tais dependências sem executar a função e sem provocar exceptions.
  • Otimização - Se um módulo não é carregado, porque vamos utilizá-lo?
  • Dependency Injection - Pois na maioria das bibliotecas (como na Require.js, abordada neste artigo) você pode definir alias para os módulos com facilidade.
  • Qualidade de código - As dependências ficam totalmente explicitas.

Legal mas..como usar AMD? Simples! Use uma biblioteca que implemente a API, e seja feliz! Pessoalmente, eu indico MUITO a Require.js, mas você é livre para escolher a que lhe parecer mais interessante.

Aliás, falando em bibliotecas, veja algumas bibliotecas que já suportam nativamente AMD:

  • jQuery
  • Dojo
  • E a maior parte das mais recentes bibliotecas JavaScript publicadas na internet.

Bom, é isso! Dúvidas? Deixe nos comentários!
